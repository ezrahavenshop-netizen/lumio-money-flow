Replit AI Prompt – Fix Mobile Hamburger Menu Scroll Glitch and Improve Mobile Navigation Experience

Issue Summary

There is a visual glitch with the mobile hamburger menu on the landing page.

When the page first loads and the hamburger menu is opened, everything appears correctly.

However, if the user scrolls down the landing page and then opens the hamburger menu, the menu becomes visually misaligned.

Based on testing and screenshots:

* Menu content appears offset.
* Navigation items shift out of place.
* Underlying page content becomes visible behind the menu.
* The overlay does not consistently cover the viewport.
* The menu appears affected by the current page scroll position.
* The experience feels broken and unpolished.

The issue appears most noticeable on mobile devices, particularly iPhone Safari, but the implementation should be reviewed across all breakpoints.

⸻

Investigation Requirements

Audit the entire mobile navigation system and determine whether the issue is caused by:

* Positioning errors (absolute vs fixed)
* Viewport height calculations (100vh)
* Scroll position inheritance
* Missing body scroll lock
* Z-index conflicts
* Mobile Safari viewport behavior
* Responsive breakpoint inconsistencies
* Overlay container rendering inside a scrolling parent

Verify whether the issue exists:

* Mobile only
* Tablet
* Desktop
* Or in the shared navigation component used across multiple breakpoints

Apply fixes globally if necessary.

⸻

Required Behavior

The hamburger menu should function as a true fullscreen navigation overlay.

Regardless of where the user is on the page:

* Top of page
* Middle of page
* Bottom of page

Opening the menu should always produce the exact same visual result.

The menu must:

* Open from the top of the viewport.
* Cover the entire screen.
* Ignore the page’s current scroll position.
* Remain perfectly aligned.
* Prevent background scrolling.
* Prevent content bleeding through the overlay.

⸻

Positioning Requirements

The menu overlay should be viewport-based rather than page-based.

Use a fullscreen overlay approach equivalent to:

position: fixed;
top: 0;
left: 0;
right: 0;
bottom: 0;
width: 100%;
height: 100vh;

or modern viewport units where appropriate:

height: 100dvh;

for improved mobile browser support.

The menu should never rely on parent container positioning.

⸻

Scroll Lock Requirements

When the menu opens:

document.body.style.overflow = 'hidden';

When the menu closes:

document.body.style.overflow = '';

or use the framework-equivalent implementation.

While the menu is open:

* The page underneath must not scroll.
* Touch gestures should not move background content.
* Scrolling should only occur within the menu if necessary.

⸻

Overlay Requirements

The overlay should:

* Cover the entire viewport.
* Block interaction with page content underneath.
* Maintain proper layering.
* Have a sufficiently high z-index.

Verify no landing page section can appear above the menu.

Check:

* Hero section
* Floating cards
* Decorative elements
* CTA buttons
* Sticky elements
* Animated backgrounds

⸻

Mobile Safari Compatibility

Specifically verify behavior on:

* iPhone Safari
* iPhone Chrome
* Android Chrome
* Android Samsung Browser

Ensure:

* No viewport jumping.
* No scroll-position offset.
* No layout shifting.
* No menu displacement after scrolling.

Use modern viewport units where necessary:

100dvh

instead of relying solely on:

100vh

where mobile browser chrome may affect sizing.

⸻

The menu should feel polished and premium.

Opening Animation
